# Progress Update

Date: 2026-04-17

This document tracks the first repair pass made after the full project review in [PROJECT_REVIEW_2026-04-17.md](PROJECT_REVIEW_2026-04-17.md).

## Fixed In This Pass

### 1. DSL type-checking no longer crashes on `use`

Problem:

- `morphogen check ...` and `morphogen run ...` crashed on `.morph` programs that used `use`.

Fix:

- Added `visit_use` to the visitor/type-checker path.
- Added missing visitor traversal methods for expression/statement nodes the checker was encountering in real examples.
- Added minimal builtin/domain symbol seeding so imported operators and common runtime symbols are recognized during type-checking.

Result:

- `python -m morphogen.cli check examples/01_hello_heat.morph` now succeeds.

### 2. `Step` and `Substep` runtime execution now use the AST shape correctly

Problem:

- Runtime execution expected `.statements` on `Step` and `Substep`, but the AST stores `.body`.

Fix:

- Updated runtime execution to iterate `step.body` and `substep.body`.

Result:

- Direct execution of parsed `step { ... }` and `substep(...) { ... }` blocks now works.

### 3. Broken `docs/DOMAINS.md` links are no longer broken

Problem:

- The repo linked repeatedly to `docs/DOMAINS.md`, but that file did not exist.

Fix:

- Added a lightweight `docs/DOMAINS.md` navigation document.

Result:

- Existing links in the README and other docs now resolve to a real file.

- Runtime test for `step { ... }`

## Second Repair Pass

### 4. CLI flow execution now matches user expectations better

Problem:

- `flow(...)` blocks were being executed via the generic initialization path.
- The CLI printed `No step blocks found, executing program once` even for real flow programs.
- `--steps` did not override `flow(steps=...)`.

Fix:

- Split `flow` blocks from plain initialization statements in `morphogen/cli.py`.
- Added explicit flow execution reporting.
- Applied CLI `--steps` overrides to flow blocks.

Result:

- `python -m morphogen.cli run examples/01_hello_heat.morph --steps 1` now reports:
  - `Executing 1 flow block(s)...`
- `--steps` now affects flow blocks as well as step blocks.

### 5. Packaging metadata is now aligned on core install requirements

Problem:

- `pyproject.toml` and `setup.py` disagreed about core dependencies.
- `pygame` was treated as a core requirement in `setup.py` even though interactive display is optional.

Fix:

- Added `pillow` to core dependencies in `pyproject.toml`.
- Added an `interactive` extra for `pygame`.
- Updated `setup.py` to mirror the same core dependency shape more closely and include the same optional extras family.

Result:

- The main packaging files no longer disagree on whether `lark` is core.
- `pygame` is now optional instead of mandatory in `setup.py`.

### 6. README drift reduced

Fix:

- Removed a stale hard-coded example count.
- Replaced stale status counts in `README.md` with softer current-surface language and a pointer to `STATUS.md`.

Result:

- The top-level README is less likely to go stale immediately after implementation changes.

## Third Repair Pass

### 7. Default `visual3d` test execution is now stable

Problem:

- `tests/test_visual3d.py` could abort the full test process because the real render path depends on an OpenGL/display stack that is not reliable in this environment.

Fix:

- Added display/backend helper coverage in `tests/test_visual3d.py`.
- Gated the true render-path tests behind `MORPHOGEN_RUN_VISUAL3D_RENDER_TESTS=1`.
- Kept the rest of the `visual3d` suite active by default.

Result:

- `pytest tests/test_visual3d.py` now completes instead of aborting.
- The project keeps broad `visual3d` coverage while making the truly fragile render tests explicit opt-in checks.

### 8. Plain `pytest` now works better in sandboxed/local environments

Problem:

- Ambient `pytest-rerunfailures` plugin loading caused an internal error before collection in this environment.

Fix:

- Disabled `rerunfailures` in the project's default pytest addopts, alongside the already-disabled PostgreSQL plugin.

Result:

- Plain `pytest ...` invocations now work without needing an extra `-p no:rerunfailures` flag.
